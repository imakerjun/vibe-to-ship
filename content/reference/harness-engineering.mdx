> **한 줄 정의**: AI 에이전트의 능력을 신뢰할 수 있는 작업으로 향하게 하는 **환경·규칙·도구 전체**를 설계하고 운영하는 접근.

## 1. 용어의 어원 — "하네스(harness)"

하네스는 원래 **말의 마구(馬具)**를 뜻합니다. 말의 힘을 유용한 일로 향하게 하는 장치죠.
AI 맥락에서는 **LLM의 능력을 신뢰할 수 있는 작업 수행으로 향하게 하는 모든 인프라·규칙·도구**를 가리킵니다.

> **핵심 공식**: `Agent = Model + Harness`
>
> — Birgitta Böckeler (Martin Fowler's bliki)

즉, **같은 모델이어도 무엇으로 감싸느냐에 따라 결과가 크게 달라집니다**. 프롬프트 한 줄을 잘 쓰는 것과, AI가 일할 **작업실 전체를 설계**하는 것은 완전히 다른 레벨의 접근입니다.

## 2. 왜 이 개념이 비개발자에게도 중요한가

**"AI에게 무엇을 시킬까"** 에서 **"AI에게 어떤 환경을 줄까"** 로 관점이 바뀌면, 아래 세 가지 문제가 자연스럽게 풀립니다.

| 자주 겪는 문제 | 하네스 관점의 해결 |
|---|---|
| "같은 질문인데 매번 답이 달라요" | 프로젝트 규칙을 **CLAUDE.md**에 한 번만 써두면 매번 같은 전제에서 출발 |
| "팀원마다 AI 활용 수준이 제각각이에요" | **커스텀 커맨드·스킬**을 팀 자산으로 공유해 출발선을 맞춤 |
| "AI가 만든 결과를 검증하기 어려워요" | **훅(Hook)** 과 **리뷰 에이전트** 로 '완료'의 정의를 코드화 |

> **오늘 여러분이 이미 하네스를 만들기 시작했습니다.** 세션 2 스텝 4에서 `/init` 명령으로 생성한 **CLAUDE.md** 가 바로 하네스의 첫 번째 조각입니다.

## 3. 하네스의 5가지 축

학술·산업계 1차 출처(Anthropic, OpenAI, Birgitta Böckeler, HumanLayer)가 공통적으로 다루는 **최소 공약수**가 아래 5가지입니다.

| # | 축 | 핵심 질문 | Claude Code에서의 구현 |
|---|---|---|---|
| 1 | **Context Engineering** | 에이전트가 **우리 세계**를 아는가? | `CLAUDE.md`, 4계층 memory hierarchy |
| 2 | **Plan-based Execution** | **합의 후** 실행하는가? | Plan Mode, 역할 분담 |
| 3 | **Token Optimization** | **길게** 일할 수 있는가? | `/compact`, 서브에이전트 격리 |
| 4 | **Quality Verification** | **완료**가 검증 가능한가? | Hooks (PreToolUse / PostToolUse), 테스트 |
| 5 | **Multi-Agent Orchestration** | **역할**이 분리돼 있는가? | Subagents, Skills |

각 축의 상세는 아래 섹션에서 구성 요소별로 이어집니다.

## 4. 구성 요소 사전 (Claude Code 기준)

### 4.1 CLAUDE.md — 프로젝트 메모리

**한 줄**: "AI에게 주는 회사 소개서". 프로젝트의 목적·기술 스택·규칙·금기사항을 기록한 마크다운 파일.

**어디에 두나**: 프로젝트 루트의 `CLAUDE.md` (또는 `~/.claude/CLAUDE.md`, `.claude/CLAUDE.md/` 등 4계층 계층 구조).

**공식 권장**: **200줄 이내**. 길수록 토큰을 많이 먹고, 변경 빈도가 높으면 캐시 효율이 떨어집니다.

**언제 필요해지나**: 같은 설명을 두 번 이상 하게 될 때. Claude Code가 "이 프로젝트는 React야" 같은 기본 정보를 매번 되묻는다면 바로 신호입니다.

**공식 문서**: [Manage Claude's memory](https://docs.claude.com/en/docs/claude-code/memory)

### 4.2 슬래시 커맨드 (Slash Commands)

**한 줄**: 자주 쓰는 프롬프트 또는 작업을 `/이름`으로 호출할 수 있는 단축 명령.

**기본 제공 명령**:
- `/init` — 프로젝트를 분석해 CLAUDE.md 초안 생성
- `/compact` — 긴 대화를 요약해서 토큰 절약
- `/review` — 최근 변경사항 코드 리뷰
- `/help` — 사용 가능한 명령 목록
- `/memory` — 메모리 파일 편집

**커스텀 명령**: `.claude/commands/{이름}.md` 파일로 만들면 `/이름`으로 호출 가능. 팀원과 공유하려면 저장소에 함께 커밋하면 됩니다.

**언제 필요해지나**: "이 프롬프트 또 써야 하는데..." 싶은 순간.

**공식 문서**: [Slash commands](https://docs.claude.com/en/docs/claude-code/slash-commands)

### 4.3 훅 (Hooks)

**한 줄**: 특정 이벤트(파일 수정 직전/직후, 대화 종료 등)가 일어나면 **자동으로 실행되는 스크립트**.

**대표 이벤트**:
- `PreToolUse` — Claude가 도구를 사용하기 직전 (예: 파일 저장 전에 포맷 검증)
- `PostToolUse` — 도구 사용 직후 (예: 저장 후 자동 커밋)
- `Stop` — 대화가 끝날 때
- `UserPromptSubmit` — 사용자가 프롬프트를 제출할 때

**비개발자 예시**: "Claude가 파일을 저장할 때마다 변경 이력을 자동으로 기록하기", "민감한 파일이 수정되면 알림 보내기".

**언제 필요해지나**: 매번 같은 확인을 반복해야 할 때. "사람이 손으로 누르는 버튼"을 "자동으로 눌리는 버튼"으로 바꿀 수 있습니다.

**공식 문서**: [Hooks guide](https://docs.claude.com/en/docs/claude-code/hooks)

### 4.4 스킬 (Agent Skills)

**한 줄**: 특정 상황에서 Claude를 안내하는 **전문 매뉴얼**. 트리거 조건이 맞으면 자동으로 참조됩니다.

**슬래시 커맨드와의 차이**:
| 커스텀 커맨드 | 스킬 |
|---|---|
| 사용자가 명시적으로 `/이름` 호출 | 상황에 맞으면 Claude가 자동 참조 |
| 단일 프롬프트 실행에 가까움 | 다단계 워크플로우 기술 가능 |

**언제 필요해지나**: "이 영역에서는 이런 순서로, 이 파일들을 이 방식으로 다룬다"처럼 **도메인별 규칙** 이 굳어졌을 때.

**공식 문서**: [Skills](https://docs.claude.com/en/docs/claude-code/skills)

### 4.5 MCP — Model Context Protocol

**한 줄**: Claude Code가 **외부 서비스**(Slack, Notion, GitHub, 데이터베이스 등)와 직접 대화할 수 있게 해주는 표준 프로토콜.

**비개발자에게 유용한 MCP 예시**:
- Notion MCP — Claude가 내 Notion 페이지를 직접 읽고 쓰기
- Slack MCP — 팀 채널에 자동 알림 보내기
- Google Drive MCP — 구글 드라이브 파일 읽기
- GitHub MCP — 이슈 조회, 코멘트 작성

**언제 필요해지나**: "Claude가 알면 좋겠는데, 내 컴퓨터에 없는 정보"가 생겼을 때.

**공식 문서**: [Model Context Protocol](https://docs.claude.com/en/docs/claude-code/mcp)

### 4.6 서브에이전트 (Subagents)

**한 줄**: **특정 역할** 을 전담하는 AI 비서를 여러 명 두는 것. 조사용, 계획용, 구현용, 리뷰용을 분리합니다.

**왜 쪼개나**: 한 에이전트가 너무 많은 일을 하면 컨텍스트가 오염돼 결과가 나빠집니다. 사람 조직도 기획·디자인·개발·QA를 나누는 것과 같은 이유입니다.

**언제 필요해지나**: 복잡한 작업에서 "AI가 스스로 검증까지 해버려서 신뢰가 안 갈 때". 구현과 검증을 분리하면 객관성이 살아납니다.

**공식 문서**: [Subagents](https://docs.claude.com/en/docs/claude-code/sub-agents)

### 4.7 Plan Mode & Permissions

**한 줄**: 실행 전에 계획을 먼저 보여주거나, 위험한 작업은 승인 받고 하도록 **권한을 설정**하는 기능.

**권한 모드**:
- `plan` — 실행 없이 계획만 세움 (검토용)
- `acceptEdits` — 파일 편집은 자동 허용
- `dontAsk` — 확인 없이 실행 (익숙해진 뒤에만)

**언제 필요해지나**: "AI가 뭔가 위험한 일을 할까봐 불안할 때". Plan 모드로 먼저 검토한 뒤 실행하면 안전합니다.

**공식 문서**: [Permission modes](https://docs.claude.com/en/docs/claude-code/settings)

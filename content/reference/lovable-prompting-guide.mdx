# Lovable 프롬프트 작성 가이드

> 원문: [docs.lovable.dev/prompting/prompting-one](https://docs.lovable.dev/prompting/prompting-one)
>
> 이 문서는 Lovable 공식 가이드 **"Prompting best practices"** 의 한국어 번역본입니다.
> 본 가이드의 핵심 메시지는 다음과 같습니다.

<Callout type="info" emoji="🎯">
**핵심 요약**
**프롬프트를 쓰기 전에 먼저 계획하고, 컴포넌트 단위로 만들고, 실제 콘텐츠를 사용하고, 디자인 용어(buzzwords)를 활용하세요.** 이렇게 하면 일관성 있고 높은 품질의 결과를 얻을 수 있습니다.
</Callout>

## 특별 기법: Lovable에게 질문을 유도하기

프롬프트 끝에 다음 문장을 덧붙이면 Lovable이 스스로 부족한 정보를 찾아서 되물어봅니다.

```
내가 이 기능에서 원하는 것과 머릿속에 그리고 있는 모습을 완전히
이해하기 위해 필요한 질문이 있다면 전부 해줘.
```

**왜 효과적인가:** Lovable이 코드를 작성하기 *전에* 빈틈을 먼저 찾아냅니다. 혼자서는 생각하지 못했을 인사이트를 얻는 경우가 많습니다.

**추천 사용 환경:** Plan 모드에서 이 기법을 사용하면 가장 효과적입니다.

**기대 효과:** 결과물이 의도와 더 가까워지고, 반복 수정 횟수가 줄어듭니다.

## Phase 1 — 기초 다지기

### 1.1 프롬프트 작성 전에 먼저 계획하라

Lovable에 요청하기 전에 **무엇을 만들지 먼저 정의**하세요.

아래 4가지 질문에 답할 수 있어야 합니다.

- **무엇을 만드는가?** (이 제품/기능이 무엇인가)
- **누구를 위한 것인가?**
- **왜 사용하는가?** (어떤 문제를 해결하는가)
- **가장 중요한 한 가지 행동은 무엇인가?** (사용자가 꼭 해야 하는 핵심 액션)

> **철학**: "모호한 아이디어는 모호한 결과물을 낳는다. 명확한 사고가 명확한 결과를 만든다."

**예시 프롬프트**

```
Gen Z 프리랜서를 대상으로 한 가계부 앱의 원페이지 사이트를 만들어줘.
핵심 CTA는 "더 스마트하게 저축 시작하기"야.
큰 글자와 강렬한 색상을 사용한 대담하고 표현력 있는 미감을 목표로 해.
```

### 1.2 사용자 여정을 시각적으로 그려라

디자인은 **화면 하나하나가 아니라, 화면 사이의 전환**으로 작동합니다.

사용자 여정을 그릴 때 아래 질문을 떠올리세요.

- 사용자가 **가장 먼저 보는 것**은 무엇인가?
- 어떻게 **신뢰**를 쌓는가?
- 어떻게 **행동할 자신감**을 주는가?
- 그 행동은 **어디로 이어지는가**?

> **핵심 인사이트**: "Hero → Features → CTA"처럼 단순한 3단계 스케치만 그려도 프롬프트 효과가 **10배** 좋아집니다.

> **철학**: "블록을 쌓는 게 아니다. 행동을 유도하는 것이다. 모든 섹션에는 존재 이유와, 다음 섹션으로 이끄는 이유가 있어야 한다."

### 1.3 디자인 방향을 먼저 정하라

시각 언어는 **장식이 아니라 기반**입니다.

**디자인 방향 예시**

- Calm and elegant (차분하고 우아한)
- Bold and disruptive (대담하고 파격적인)
- Premium and sleek (고급스럽고 세련된)

**적용 방법**: 스타일을 프롬프트에 직접 명시하세요. buzzwords, 톤 묘사, UI 패턴을 함께 사용합니다. 자주 쓰는 조합은 **재사용 가능한 "스타터 스타일 프롬프트"** 로 만들어두면 좋습니다.

> **철학**: "좋은 디자인은 프롬프트로 도달하는 게 아니다. 좋은 디자인에서 출발해서 프롬프트를 쓴다."

**예시 프롬프트**

```
차분한 웰니스 스타일의 디자인을 사용해줘.
부드러운 그라데이션, 차분한 어스톤, 둥근 모서리, 여유로운 여백을 사용해.
폰트는 "Inter"를 써줘.
전체 톤은 부드럽고 안심되는 느낌이어야 해.
```

## Phase 2 — 시스템적으로 생각하기

### 2.1 페이지가 아니라 컴포넌트 단위로 프롬프트하라

전체 페이지를 한 번에 만들지 말고 **모듈 단위로 쌓으세요**.

- **전체 페이지 요청** → 정밀도가 떨어지고 결과가 흐릿함
- **컴포넌트 단위 요청** → 명료함, 통제, 유연성

**컴포넌트 방식의 장점**

- 블록 하나씩 따로 수정할 수 있음
- 여러 페이지에 걸쳐 확장 가능한 디자인 사고
- 컴포넌트 재사용과 변형이 쉬움

> **철학**: "전체 페이지 프롬프트는 소음을 얻는다. 섹션 단위 프롬프트는 신호를 얻는다." (A full-page prompt gets you noise. A section-based prompt gets you signal.)

**예시 프롬프트**

```
글래스모피즘 효과를 적용한 플로팅 메뉴 바를 만들어줘.
Home, Search, Music, Favorites, Add, Profile, Settings 아이콘을 포함해.
부드러운 플로팅 애니메이션과 매끄러운 호버 인터랙션도 추가해.
```

### 2.2 실제 콘텐츠로 디자인하라

**플레이스홀더("lorem ipsum", "기능 1/기능 2" 등)를 피하세요.**

모델은 **구조와 의도**에 반응하는데, 의도를 가장 잘 보여주는 것이 **실제 단어**입니다.

**콘텐츠 가이드**: 최종 카피가 아직 확정되지 않았더라도, **실제 메시지를 반영한 문구**를 써주세요.

**두 가지 이점**

1. Lovable의 레이아웃과 여백 생성 품질이 개선됨
2. 디자인 결정이 더 일찍 드러남 (예: 헤드라인의 줄바꿈, CTA 동사 구조)

> **철학**: "디자인은 제약을 사랑한다. 실제 콘텐츠가 올바른 제약을 만든다."

**예시 프롬프트**

```
Hero 섹션 헤드라인: "차분하게 디자인하세요."
서브 텍스트: "Lovable로 스트레스를 구조로 바꾸세요."
CTA: "무료로 만들기 시작하기"
카피 중심 레이아웃, 넉넉한 수직 여백을 사용해.
```

### 2.3 원자적(Atomic) UI 용어로 말하라

Lovable은 **원자적 UI 언어**를 가장 잘 이해합니다.

**원자적 용어 예시**: Cards, badges, toggles, chips, form fields, dropdowns, modals, toasts

**적용 전략**: 모호한 요청을 **구체적인 UI 패턴** 으로 바꾸세요.

**복잡도 쌓기**: 기본 원자(card)로 시작한 뒤 점진적으로 더하기 (badges → hover 효과 → 툴팁)

> **철학**: "부분이 작을수록 응답이 똑똑해진다." (The smaller the part, the smarter the response.)

**예시 프롬프트**

```
사용자 프로필 사진, 이름, Follow 버튼이 있는 카드를 만들어줘.
인증된 사용자에게는 배지를 붙이고,
배지에 마우스를 올리면 툴팁이 나오도록 해줘.
```

### 2.4 Buzzwords로 미감을 조율하라

**Buzzwords는 시각적 결과를 조정하는 프롬프트 파라미터** 역할을 합니다.

**권장 buzzwords**

- `minimal` (미니멀한)
- `expressive` (표현력 있는)
- `cinematic` (시네마틱한)
- `playful` (장난스러운)
- `premium` (고급스러운)
- `developer-focused` (개발자 친화적인)

**사용 전략**

- 프롬프트 **앞쪽**에 배치하세요
- 가능하면 **모든 섹션마다** 사용하세요
- 섹션마다 다른 buzzword를 섞어도 됩니다 (예: "대담하고 파격적인" hero → "차분하고 안심되는" pricing)

> **철학**: "디자인은 구조만이 아니다. 톤이다. Buzzwords가 톤을 정의한다."

**예시 프롬프트**

```
premium하고 cinematic한 느낌의 랜딩 페이지 hero를 디자인해줘.
레이어드된 깊이감, 반투명 표면, 부드러운 모션 블러,
그리고 헤드라인과 배경 사이의 극적인 대비를 사용해.
```

## Phase 3 — 정교하게 만들기

### 3.1 레이아웃에 프롬프트 패턴을 사용하라

**구조화되고 반복 가능한 프롬프트 템플릿**을 사용하세요.

**공통 구조**: `헤더 → 본문 → 액션` (디자인에 따라 변형 가능)

**적용 방법**: 자주 쓰는 컴포넌트(feature grid, testimonial slider, pricing table)를 **나만의 프롬프트 라이브러리**로 정리해두세요.

> **철학**: "Lovable은 구조화되고, 범위가 명확하며, 예측 가능한 프롬프트 언어에 가장 잘 반응한다."

**예시 프롬프트**

```
가운데 정렬된 헤드라인과 가로로 나란히 놓인 카드 3개로 구성된
feature 섹션을 만들어줘.
각 카드는 상단 아이콘, 헤드라인, 짧은 설명을 포함해.
카드에는 부드러운 그림자와 hover 시 살짝 떠오르는 효과를 넣어줘.
```

### 3.2 URL로 시각 자료를 추가하라

Lovable은 **URL을 통해 영상 임베드**를 지원합니다.

프롬프트에 아래 요소를 명시하세요.

- **배치 위치** (예: Hero 아래, feature 카드 안)
- **스타일** (둥근 모서리, 자동 재생, 음소거)
- **맥락** (왜 이 영상을 넣는가)

**예시 프롬프트**

```
Midjourney의 제품 데모 영상을 임베드해줘. 이 URL을 사용해:
https://cdn.midjourney.com/video/cb84f296-92a0-4a37-a0e3-1c9c95299488/0.mp4
feature 섹션 아래에 full-width 카드로 배치하고, 부드러운 그림자를 추가해.
```

### 3.3 Edit 버튼으로 맥락을 겹쳐가며 다듬어라

**Edit 버튼**은 특정 레이어/요소에만 집중해서 반복 수정할 수 있게 해줍니다.

**사고 방식**: 수정을 *완전한 교체* 가 아니라 **디자인 오버라이드**로 대하세요.

**실제 예시**

- CTA 버튼을 선택해서 **텍스트만** 변경
- 카드 레이아웃을 조정하되 **타이포그래피는 건드리지 않기**

**정밀한 언어 사용**: 모호한 표현 대신 `replace`, `update`, `adjust` 같은 구체적 동사를 쓰세요.

**수정 예시 프롬프트**

```
CTA 버튼 텍스트를 "Get Started"로 바꾸고, 가로 padding을 24px로 늘려줘.
배경색과 폰트는 그대로 유지해.
```

> **철학**: "편집 기능은 프로젝트를 깔끔하고 모듈화된 상태로 유지시키고, 프롬프트 과부하를 줄이며, 워크플로우를 가속화하고, 실험에 대한 자신감을 높여준다."

## Phase 4 — 반복하고 배포하기

### 4.1 Lovable Cloud를 염두에 두고 설계하라

**Lovable Cloud**는 인증, 데이터베이스, 저장소, edge function을 관리합니다.

디자인 초기부터 아래를 고려하세요.

- **인증 로직**: 로그인한 사용자와 하지 않은 사용자에게 각각 무엇이 보이는가?
- **동적 콘텐츠**: 사용자 데이터, 게시물, 항목, 지표를 어디에서 불러오는가?
- **상태 처리**: 빈 데이터, 로딩 중, 실패 상황을 어떻게 보여주는가?

**구현 타이밍**: 디자인 단계에서 백엔드가 반드시 필요하진 않지만, **UI 구조는 실제 기능을 미리 예상**하고 설계해야 합니다.

**예시 프롬프트**

```
Cloud를 통해 로그인한 사용자라면 우측 상단에 프로필 이미지와 이름을 표시해줘.
로그인하지 않았다면 "Log In" 버튼을 보여주고, 클릭하면 인증 화면으로 이동해.
```

### 4.2 버전 관리를 습관으로 삼아라

**자동 저장은 편리하지만, 의도적인 반복을 대신하지는 못합니다.**

**권장 실천**

- **마일스톤 단위로 사고하기**: 레이아웃 확정 → 콘텐츠 추가 → 로직 연결
- **프롬프트 노트**로 변경 사항 기록: 예) "CTA 텍스트 변경 및 카드 간격 조정"
- **중요한 변경 전에는 먼저 미리보기**, **위험한 반복 전에는 먼저 복제**

> **철학**: "자동 저장이 자동 정리를 의미하지는 않는다. 의도를 가지고 만들고, 자각하며 반복하라."

**습관 예시**

```
큰 변경을 하기 전에, 현재 버전을 복제해두고 라벨을 붙여라.
작은 비용, 큰 안전망.
```

## 핵심 원칙 12가지 요약

| # | 원칙 | 한 줄 |
|---|------|------|
| 1 | **Plan before prompting** | 프롬프트 전에 먼저 생각을 정리하라 |
| 2 | **Map user journey** | 사용자 여정을 시각적으로 그려라 |
| 3 | **Design direction first** | 시각 방향을 먼저 정하라 |
| 4 | **Component, not page** | 컴포넌트 단위로 만들어라 |
| 5 | **Real content** | 실제 콘텐츠로 디자인하라 |
| 6 | **Atomic UI vocabulary** | 원자적 UI 용어로 말하라 |
| 7 | **Use buzzwords** | Buzzwords로 미감을 조율하라 |
| 8 | **Structured prompt patterns** | 구조화된 프롬프트 패턴을 써라 |
| 9 | **URLs for visuals** | 필요할 때 URL로 시각 자료를 추가하라 |
| 10 | **Edit button for refinement** | Edit 버튼으로 정밀하게 다듬어라 |
| 11 | **Design with Cloud in mind** | Cloud 아키텍처를 염두에 두고 설계하라 |
| 12 | **Intentional versioning** | 의도적으로 버전을 관리하라 |

## 워크숍에서 활용하기

이 가이드는 **세션 2 스텝 1 (업무 파악 + Lovable MVP)** 에서 프롬프트를 작성할 때 참고하면 좋습니다.

특히 아래 세 가지는 반드시 적용해보세요.

1. **Planning first** — What/Who/Why/CTA를 프롬프트 첫 문장에 명시

# 스텝 1: 업무 파악 + Lovable MVP

<Callout type="info">
**세션 1과 다른 점은 딱 하나** — 프롬프트를 직접 작성합니다.
결정할 것은 3가지뿐입니다: (1) 앱 주제, (2) 핵심 기능 1가지, (3) 저장할 데이터.
이 3가지만 정하면 나머지는 세션 1과 동일합니다.
</Callout>

## 어떤 앱을 만들까?

**거창할 필요 없습니다.** 아래 질문에 답해보세요.

> 이번 주에 **엑셀이나 수작업으로 반복한 일**이 있나요?
> 그걸 앱으로 바꾸면 어떤 모습일까요?

### 스코핑 템플릿

> **5분 아이디어 선정법**: 2분 각자 메모 → 1분 돌아가며 발표 → 1분 투표 → 최다 득표 선택

조별로 아래를 채워주세요. **오늘 안에 가능한 범위**에 집중합니다.

```
[앱 이름]:

누가 쓰나: (나? 팀원? 고객?)
뭘 넣나: (입력하는 데이터)
뭘 보나: (화면에 나오는 결과)
저장할 것: (DB에 남길 데이터)
오늘 우선 만들 것: ___만 먼저 만든다
```

### 아이디어 예시

이전 교육에서 동료분들이 실제로 만든 것들입니다. 참고해서 본인 업무에 맞는 아이디어를 떠올려보세요.

| 유형 | OK금융그룹 실제 사례 | 오늘 만들 수 있는 버전 |
|------|---------------------|----------------------|
| **업무 도구/유틸리티** | 회의실 예약 시스템 (14개 실시간 현황), 법무사보수 자동계산기 (취득세·중개수수료), D-day 스케줄 위젯 | 회의실 예약 현황판, 각종 계산기, 일정 D-day 관리 페이지 |
| **고객 응대/상담** | 응대스크립트 웹 뷰어 (엑셀 → 웹 조회+복사), 고객응대 답변가이드 챗봇 | 자주 쓰는 스크립트/안내문을 검색하고 클립보드에 복사하는 웹 뷰어 |
| **HR/인사** | HR 평가자 자동매핑 (이동발령 시 자동 변경), 위탁교육 관리 (PDF→신청자 정보 추출) | 평가 점수 입력 → 등급 자동 계산, 교육 신청 현황 관리 페이지 |
| **자산/채권 현황** | IB금융본부 자산현황 대시보드 (KPI·포트폴리오 차트), 이자상환 스케줄 관리 | 팀이 매주 확인하는 수치를 한 화면에 모아보는 대시보드 |
| **금융 데이터 조회** | 업무원가율 조회 (엑셀 업로드→DB→웹 조회), 저축은행 금리 모니터링 | 자주 보는 데이터를 직접 입력하거나 붙여넣어 조회하는 화면 |
| **보고서/문서 도구** | IB Pre보고 자동 작성, DART 감사보고서 수집·분석 초안 | 정해진 양식에 값을 채워 넣으면 보고 초안을 만들어주는 폼 |
| **컴플라이언스/점검** | 공시 알리미 (법령 기준 자동 분석), 채권관리 컴플라이언스, 법인카드 점검 | 체크 항목 목록을 입력하면 완료 여부를 표시해주는 점검 체크리스트 |

<Callout type="warning">
**범위를 좁히세요!** "전부 다 만들기"보다 "핵심 1가지를 완성"하는 게 훨씬 낫습니다.

**좋은 범위 예시:**
- "회의실 예약 전체 시스템" ✕ → "우리 팀 회의실 오늘 예약 현황을 한눈에 보는 페이지" ✓
- "HR 평가 자동화 시스템" ✕ → "평가 점수를 입력하면 등급을 자동 계산해주는 도구" ✓
- "자산 관리 플랫폼" ✕ → "매주 정리하는 핵심 KPI 수치 3개를 카드로 보여주는 대시보드" ✓
</Callout>

## Lovable MVP 만들기

세션 1의 스텝 1과 동일한 흐름입니다. 이번에는 **프롬프트를 직접 작성**합니다. Lovable 공식 [Prompt Engineering 가이드](https://docs.lovable.dev/prompting)의 권장 흐름을 그대로 따릅니다.

### 1단계: 계획(Plan) — 4가지를 먼저 정합니다

Lovable 가이드는 코딩을 시작하기 전에 네 가지 질문에 답하라고 권장합니다. 스코핑 템플릿에서 이미 정한 내용을 여기 한 번 더 적어보세요.

| 질문 | 내 답 |
|---|---|
| **What** — 무엇을 만드는가 | `한 줄로 요약한 앱 정체성` |
| **Who** — 누가 쓰는가 | `구체적인 사용자 역할, 부서` |
| **Why** — 왜 쓰는가 | `이 앱이 해결하는 업무 문제` |
| **Main CTA** — 사용자가 가장 자주 누를 버튼/행동 | `예: "오늘 현황 확인", "신청 제출"` |

### 2단계: 첫 프롬프트 템플릿

Lovable 공식 가이드가 제시하는 기본 구조는 **`[제품/기능 정의] + [대상 사용자] + [시각적 톤] + [주요 CTA]`** 입니다. 아래 템플릿을 그대로 채워 넣으세요.

```
{한 줄 제품 정의}를 만들어줘. {대상 사용자 설명}을 위한 앱이야.
주요 행동은 "{Main CTA}"이고, 이 버튼이 가장 눈에 띄어야 해.

핵심 기능 (오늘은 1개만 먼저):
- {기능 1을 동사로 시작해 구체적으로}

화면 구성:
- 상단: {헤더에 보이는 것}
- 본문: {카드/리스트/폼 중 어떤 컴포넌트}
- 하단: {있다면}

샘플 데이터 (실제 업무 예시로):
- {레코드 1}
- {레코드 2}
- {레코드 3}

디자인 톤: {minimal / premium / playful / calm 중 하나} + {색상 키워드}
반응형: 모바일까지 고려
```

<Callout type="info">
**왜 이 템플릿인가**

Lovable 공식 가이드의 세 가지 원칙을 그대로 반영한 것입니다.
1. **Planning first** — What/Who/Why/CTA를 프롬프트 첫 문장에 명시합니다.
2. **Component-based** — "카드", "리스트", "폼" 같은 원자적 UI 용어를 사용합니다. "A full-page prompt gets you noise. A section-based prompt gets you signal."
3. **Real content, not lorem ipsum** — 샘플 데이터는 실제 업무에서 나올 법한 값으로 씁니다.
</Callout>

### 3단계: Do & Don't

Lovable 가이드가 반복해서 강조하는 베스트 프랙티스입니다.

**DO**
- **한 섹션씩 요청하세요** — 첫 프롬프트로 전체 앱을 한 번에 만들려 하지 말고, 핵심 기능 1개만 먼저 만들고 후속 채팅으로 추가합니다.
- **실제 콘텐츠를 넣으세요** — "상품명", "사용자1" 같은 플레이스홀더 대신 실제 업무에서 쓸 값(고객사명, 품목명, 날짜)을 예시로 넣으면 결과가 훨씬 구체적입니다.
- **디자인 버즈워드를 쓰세요** — "예쁘게" 대신 `minimal`, `premium`, `calm`, `playful` 같은 공식 가이드 권장 키워드를 사용합니다.
- **편집 기능을 쓰세요** — 부분 수정이 필요하면 전체 재작성을 요청하지 말고 "이 카드의 배경만 진한 파란색으로" 같이 특정 요소만 지적합니다.

**DON'T**
- 한 프롬프트에 기능 5개 이상 나열
- "좋게 만들어줘", "깔끔하게" 같은 모호한 표현
- `lorem ipsum` 같은 더미 텍스트로 레이아웃만 확인
- 레이아웃 논리 없이 요소들을 나열하기

## 확인 포인트

> 앱의 핵심 기능 1개가 프리뷰에서 동작하면 스텝 1 완료입니다.

## 원하는 결과가 안 나왔을 때

<Callout type="info">
- 화면 오른쪽 프리뷰를 보면서 구체적으로 지적하세요: "버튼이 없어. 상단에 추가해줘"
- 한 번에 전부 고치려 하지 마세요. 가장 중요한 것 1가지만 요청하세요
- 3번 이상 수정했는데도 안 되면 새 채팅을 시작하는 것이 더 빠릅니다
</Callout>

## 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| Lovable이 전혀 다른 앱을 만들었다 | 프롬프트가 너무 추상적 | 스코핑 템플릿의 '핵심 기능'을 더 구체적으로 써서 새 채팅에서 다시 시도 |
| "이해할 수 없는 요청" 에러가 뜬다 | 프롬프트가 너무 길거나 모순됨 | 핵심 기능 1개만 남기고 줄이기 |
| 생성이 중간에 멈췄다 | 네트워크 불안정 또는 Lovable 서버 과부하 | 잠시 후 재시도 |
| 프리뷰에서 빨간 텍스트(에러 화면)가 보인다 | 생성된 코드에 오류 포함 | Lovable에게 "에러가 나고 있어, 확인해줘"라고 요청 |
| 만들어졌는데 디자인이 엉망이다 | 기능보다 레이아웃이 먼저 틀어짐 | 기능을 먼저 확인하고, 디자인은 나중에 "카드 형태로 깔끔하게 바꿔줘"로 별도 요청 |

<Callout type="warning">
**5분 룰** — 5분 이상 같은 문제로 막혀있다면 옆 동료나 강사에게 도움을 요청해주세요. 혼자 끙끙대는 것보다 빠릅니다.
</Callout>

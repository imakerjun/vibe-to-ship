# 스텝 2: Lovable 첫 프롬프트

<Callout type="info">
**이 스텝의 목표** — 스텝 1에서 정한 아이디어를 **Lovable에 넣을 첫 프롬프트**로 옮기고, 핵심 기능 1개가 프리뷰에서 동작하는 MVP를 만듭니다.
</Callout>

세션 1의 스텝 1과 동일한 흐름입니다. 이번에는 **프롬프트를 직접 작성**합니다.
Lovable 공식 [Prompt Engineering 가이드](https://docs.lovable.dev/prompting)의 권장 흐름을 그대로 따릅니다.

<Callout type="default">
**영어가 부담되면 한국어 번역본을 보세요**
공식 가이드를 한국어로 풀어 쓴 [Lovable 프롬프트 작성 가이드](/reference/lovable-prompting-guide)를 참고 자료에 준비해뒀습니다. 12가지 핵심 원칙과 예시 프롬프트를 그대로 번역해두었습니다.
</Callout>

## 1단계: 계획(Plan) — 4가지를 먼저 정합니다

Lovable 가이드는 코딩을 시작하기 전에 네 가지 질문에 답하라고 권장합니다. 스텝 1에서 이미 정한 내용을 여기 한 번 더 적어보세요.

| 질문 | 내 답 |
|---|---|
| **What** — 무엇을 만드는가 | `한 줄로 요약한 앱 정체성` |
| **Who** — 누가 쓰는가 | `구체적인 사용자 역할, 부서` |
| **Why** — 왜 쓰는가 | `이 앱이 해결하는 업무 문제` |
| **Main CTA** — 사용자가 가장 자주 누를 버튼/행동 | `예: "오늘 현황 확인", "신청 제출"` |

## 2단계: 첫 프롬프트 템플릿

Lovable 공식 가이드가 제시하는 기본 구조는 **`[제품/기능 정의] + [대상 사용자] + [시각적 톤] + [주요 CTA]`** 입니다. 아래 템플릿을 그대로 채워 넣으세요.

```
{한 줄 제품 정의}를 만들어줘. {대상 사용자 설명}을 위한 앱이야.
주요 행동은 "{Main CTA}"이고, 이 버튼이 가장 눈에 띄어야 해.

핵심 기능 (오늘은 1개만 먼저):
- {기능 1을 동사로 시작해 구체적으로}

화면 구성:
- 상단: {헤더에 보이는 것}
- 본문: {카드/리스트/폼 중 어떤 컴포넌트}
- 하단: {있다면}

샘플 데이터 (실제 업무 예시로):
- {레코드 1}
- {레코드 2}
- {레코드 3}

디자인 톤: {minimal / premium / playful / calm 중 하나} + {색상 키워드}
반응형: 모바일까지 고려
```

<Callout type="info">
**왜 이 템플릿인가**

Lovable 공식 가이드의 세 가지 원칙을 그대로 반영한 것입니다.
1. **Planning first** — What/Who/Why/CTA를 프롬프트 첫 문장에 명시합니다.
2. **Component-based** — "카드", "리스트", "폼" 같은 원자적 UI 용어를 사용합니다. "A full-page prompt gets you noise. A section-based prompt gets you signal."
3. **Real content, not lorem ipsum** — 샘플 데이터는 실제 업무에서 나올 법한 값으로 씁니다.
</Callout>

## 3단계: 템플릿이 어려우면 Claude에게 부탁하세요

빈칸을 하나씩 채우는 게 부담스럽다면, 아래 블록을 그대로 복사해 Claude에게 붙여넣으세요. **2단계의 Lovable 템플릿**과 **스프레드시트 기획 내용**을 한 번에 전달합니다. Claude Code, Claude.ai, Claude 앱 어디서나 사용할 수 있습니다.

```
아래는 Lovable 공식 프롬프트 가이드의 기본 구조입니다.
이 구조에 맞춰서 첫 번째 빌드 프롬프트를 작성해주세요.
필요한 정보가 있다면 저에게 질문해주세요.

[Lovable 프롬프트 기본 구조]
{한 줄 제품 정의}를 만들어줘. {대상 사용자 설명}을 위한 앱이야.
주요 행동은 "{Main CTA}"이고, 이 버튼이 가장 눈에 띄어야 해.

핵심 기능 (오늘은 1개만):
- {기능 1을 동사로 시작해 구체적으로}

화면 구성:
- 상단: {헤더에 보이는 것}
- 본문: {카드/리스트/폼 중 어떤 컴포넌트}
- 하단: {있다면}

샘플 데이터 (실제 업무 예시로):
- {레코드 1}
- {레코드 2}
- {레코드 3}

디자인 톤: {minimal / premium / playful / calm} + {색상 키워드}
반응형: 모바일까지 고려

---

[제가 작성한 기획 아이디어]
// 스프레드시트에 작성한 내용을 붙여넣어주세요.
- 앱 이름:
- 어떤 업무를 편하게 만들어 주나요?:
- 누가 쓰나요?:
- 화면에 뭐가 보이면 좋을까요? (출력):
- 어떤 정보를 넣을 수 있어야 하나요? (입력):
- 다음에 또 켰을 때 남아있어야 하는 것은? (저장):
- 오늘 가장 먼저 만들 딱 하나의 기능:
```

Claude가 돌려준 프롬프트를 그대로 복사해서 Lovable에 붙여넣으면 됩니다.

<Callout type="default">
**Tip** — 결과물이 마음에 들지 않으면 "기능을 더 단순하게" 또는 "디자인 톤을 `minimal`로 바꿔줘"처럼 **하나씩만** 수정 요청하세요. Lovable에 붙여넣기 전에 한 번 더 읽어보고 실제 업무 용어로 다듬는 게 좋습니다.
</Callout>

## 4단계: Do & Don't

Lovable 가이드가 반복해서 강조하는 베스트 프랙티스입니다.

**DO**
- **한 섹션씩 요청하세요** — 첫 프롬프트로 전체 앱을 한 번에 만들려 하지 말고, 핵심 기능 1개만 먼저 만들고 후속 채팅으로 추가합니다.
- **실제 콘텐츠를 넣으세요** — "상품명", "사용자1" 같은 플레이스홀더 대신 실제 업무에서 쓸 값(고객사명, 품목명, 날짜)을 예시로 넣으면 결과가 훨씬 구체적입니다.
- **디자인 버즈워드를 쓰세요** — 버즈워드는 AI가 학습한 **디자인 스타일을 한 단어로 호출하는 키워드**입니다. "예쁘게" 같은 모호한 말 대신 `minimal`(군더더기 없는), `premium`(고급스러운), `calm`(차분한), `playful`(경쾌한) 같은 공식 가이드 권장 키워드를 쓰면 톤이 훨씬 일관되게 나옵니다.
- **편집 기능을 쓰세요** — 부분 수정이 필요하면 전체 재작성을 요청하지 말고 "이 카드의 배경만 진한 파란색으로" 같이 특정 요소만 지적합니다.

**DON'T**
- 한 프롬프트에 기능 5개 이상 나열
- "좋게 만들어줘", "깔끔하게" 같은 모호한 표현
- `lorem ipsum` 같은 더미 텍스트로 레이아웃만 확인
- 레이아웃 논리 없이 요소들을 나열하기

> 앱의 핵심 기능 1개가 프리뷰에서 동작하면 스텝 2 완료입니다.

## 원하는 결과가 안 나왔을 때

<Callout type="info">
- 화면 오른쪽 프리뷰를 보면서 구체적으로 지적하세요: "버튼이 없어. 상단에 추가해줘"
- 한 번에 전부 고치려 하지 마세요. 가장 중요한 것 1가지만 요청하세요
- 3번 이상 수정했는데도 안 되면 새 채팅을 시작하는 것이 더 빠릅니다
</Callout>

## 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| Lovable이 전혀 다른 앱을 만들었다 | 프롬프트가 너무 추상적 | 스코핑 템플릿의 '핵심 기능'을 더 구체적으로 써서 새 채팅에서 다시 시도 |
| "이해할 수 없는 요청" 에러가 뜬다 | 프롬프트가 너무 길거나 모순됨 | 핵심 기능 1개만 남기고 줄이기 |
| 생성이 중간에 멈췄다 | 네트워크 불안정 또는 Lovable 서버 과부하 | 잠시 후 재시도 |
| 프리뷰에서 빨간 텍스트(에러 화면)가 보인다 | 생성된 코드에 오류 포함 | Lovable에게 "에러가 나고 있어, 확인해줘"라고 요청 |
| 만들어졌는데 디자인이 엉망이다 | 기능보다 레이아웃이 먼저 틀어짐 | 기능을 먼저 확인하고, 디자인은 나중에 "카드 형태로 깔끔하게 바꿔줘"로 별도 요청 |

<Callout type="warning">
**5분 룰** — 5분 이상 같은 문제로 막혀있다면 옆 동료나 강사에게 도움을 요청해주세요. 혼자 끙끙대는 것보다 빠릅니다.
</Callout>

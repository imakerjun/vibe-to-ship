# 스텝 6: 데모타임 + 회고

## 데모타임

각 조가 만든 앱을 공유합니다.

<Callout type="default">
**오늘 아침에는 아이디어만 있었습니다. 지금은 동작하는 URL이 있습니다.**
완성도가 아닌, **이 변화 자체**가 데모의 핵심입니다.
</Callout>

### 데모 형식 — 3단계 구조

참가자가 많아 한 명씩 전체 발표는 어렵습니다. 대신 **"부담 없이 보여주기 → 낯선 사람에게 보여주기 → 전체 하이라이트"** 3단계로 진행합니다.

#### 1단계: 조 내부 쇼케이스 (10분)

조원끼리 돌아가며 자기 화면을 보여줍니다. **1인당 2분**, 아래 3가지만 짧게 공유합니다.

1. **Before**: "우리 팀은 ___를 매번 수작업으로 했어요"
2. **After**: 내 화면에서 핵심 기능 1개를 직접 시연
3. **발견**: "만들면서 알게 된 건 ___였어요"

> 가장 편한 사람(조원)에게 먼저 말해보는 단계입니다. 잘 만들었는지는 중요하지 않습니다.

#### 2단계: 2개조 합동 갤러리 워크 (15분)

짝 조끼리 합칩니다: **1-2조 / 3-4조 / 5-6조 / 7-8조**.

- 각자 자기 자리에 **노트북을 열어둔 채** 앉아 있습니다 (= 전시)
- 나머지 사람은 **자유롭게 돌아다니며 구경**합니다 (= 관람)
- 마음에 드는 앱 앞에서 멈춰 "이거 어떻게 만들었어요?" 물어봅니다
- 절반 지점에 전시자와 관람자 역할을 바꿉니다

> 발표자-청중 구조가 아닌 **전시회 방식**이라, 내향적인 참가자도 편안하게 참여할 수 있습니다. 자연스럽게 조 간 교류도 생깁니다.

#### 3단계: 전체 하이라이트 — 30초 번개 소개 (10분)

각 조에서 **"오, 이거 신기하다"** 싶은 앱 1개씩 추천합니다. 만든 사람이 **30초 번개 소개**를 합니다.

- 손을 드는 게 아니라 **조원이 골라주는** 방식입니다 — 심리적 부담이 훨씬 낮습니다
- 30초 타이머를 띄워놓고, 딱 그 시간만 씁니다
- 8개조 × 30초 = 4분, 전환 시간 포함 10분 안에 마무리됩니다

<Callout type="info">
**왜 이 구조인가?**

인원이 많을 때 전체 발표는 "용기 있는 소수"만의 무대가 되기 쉽습니다. 이 3단계는 모든 참가자가 **최소 한 번은 자기 작품을 말로 소개**하는 경험을 하면서도, 부담은 조 내부 → 합동 → 전체 순으로 단계적으로만 올라가게 설계했습니다.
</Callout>

<Callout type="warning">
**데모가 안 될 때 대응 가이드**

기술적인 문제는 언제든 생깁니다. 당황하지 말고 아래 순서로 대응하세요.

- **배포 URL이 안 열릴 때** → Lovable 에디터의 로컬 프리뷰(오른쪽 패널)로 시연합니다. 배포가 안 됐을 뿐, 앱은 동작합니다.
- **앱이 에러 화면을 보일 때** → 에러 전/후 스크린샷을 보여주세요. "이렇게 만들었고, 이 부분을 개선해야 한다는 걸 발견했다"는 것 자체가 발표 내용입니다.
- **아예 미완성일 때** → 프롬프트 대화 내역과 설계 과정을 발표합니다. "우리가 해결하려 한 문제, 어떤 접근을 시도했는지"를 공유하면 됩니다.

완성도가 아닌 **사고 과정**이 이 워크숍의 핵심입니다.
</Callout>

## 회고

### 개인 회고

아래 3가지에 답해보세요. 조원과 돌아가며 공유합니다.

1. **Before & After**: 오늘 시작 전의 나 vs 지금의 나, 가장 달라진 점은?
2. **Best Moment**: 오늘 가장 뿌듯했거나 놀라웠던 순간은?
3. **Monday Action**: 월요일에 바로 시도해볼 수 있는 것 하나는?

### 도구 가이드

오늘 사용한 도구들의 역할을 정리합니다.

| 단계 | 도구 | 역할 |
|------|------|------|
| 아이디어 → 프로토타입 | **Lovable** | 대화로 동작하는 앱을 만든다 |
| 데이터 저장 | **Lovable Cloud** | 새로고침해도 데이터가 유지된다 |
| 정밀한 개선 | **Claude Code** (VS Code) | 코드 수준으로 기능을 추가·수정한다 |
| 배포 | **Lovable Publish** | 다른 사람이 접속할 수 있는 URL을 만든다 |
| 자산화 | **CLAUDE.md** + 커스텀 커맨드 | 반복 작업을 자동화하고 팀과 공유한다 |

### Q&A

<Toggle title="회사 내부 데이터를 Lovable이나 Claude에 넣어도 되나요?">
개인정보·비밀 데이터를 외부 AI 서비스에 직접 입력하는 것은 현재 회사 보안 정책을 먼저 확인해야 합니다. 오늘 워크숍에서는 가상/샘플 데이터로 구조만 만들고, 실제 운영 시 사내 폐쇄망 환경이나 API 방식으로 전환하는 것이 현실적인 경로입니다.
</Toggle>

<Toggle title="무료로 계속 쓸 수 있나요? 비용은 어떻게 되나요?">
**Lovable**은 무료 플랜(월간/일간 크레딧 단위로 제공)과 여러 단계의 유료 플랜이 있습니다. 무료 크레딧 한도 내에서 충분히 체험·검증이 가능합니다. 정확한 크레딧 수와 금액은 변동될 수 있어 [공식 Pricing 페이지](https://lovable.dev/pricing)를 참고하세요.

**Claude Code**는 Claude의 유료 구독 플랜(Pro / Max / Team / Enterprise)에서 사용할 수 있습니다. 무료 Claude 계정으로는 사용할 수 없습니다. 최신 플랜과 가격은 [Anthropic Pricing](https://www.anthropic.com/pricing)에서 확인하세요.

처음엔 무료 한도로 충분히 검증하고, 실무에 반복 사용한다면 유료 전환을 검토하세요. 수작업 시간 대비 ROI를 따지면 대부분 충분히 납득할 수준입니다.
</Toggle>

<Toggle title="만든 앱을 팀원들이 쓰려면 어떻게 하나요?">
Lovable에서 **Publish** 버튼을 누르면 누구나 접속할 수 있는 URL이 생깁니다. 별도 서버 설치 없이 링크만 공유하면 됩니다. 다만 Lovable 무료 플랜은 공개 URL 수에 제한이 있으니, 팀 전체가 쓰는 앱이라면 유료 플랜이나 별도 배포(Vercel 등)를 검토하세요.
</Toggle>

<Toggle title="코딩을 몰라도 나중에 유지보수할 수 있나요?">
기능 추가·수정은 Lovable 채팅으로 계속 할 수 있습니다. 다만 버그가 생기거나 복잡한 로직이 필요해지면 코드를 봐야 하는 순간이 옵니다. Claude Code를 이용하면 코드를 몰라도 "이 버튼이 동작을 안 해"라고 설명하는 것만으로 수정이 가능합니다. 완전한 무코딩은 아니지만, 개발자 의존도는 크게 낮출 수 있습니다.
</Toggle>

<Toggle title="회사 기존 시스템(그룹웨어, ERP 등)과 연동하려면 어떻게 하나요?">
두 가지 경로가 있습니다. ① 시스템이 API를 제공한다면 Claude Code로 연동 코드를 작성할 수 있습니다. ② API가 없다면 엑셀/CSV 내보내기 → 앱에 업로드하는 방식으로 "반쪽짜리 자동화"도 충분히 효과적입니다. 오늘 만든 앱 대부분이 이 방식입니다.
</Toggle>

<Toggle title="Claude Code 말고 다른 AI 코딩 도구는 없나요?">
GitHub Copilot(에디터 자동완성), Cursor(AI 네이티브 에디터), v0(Vercel, UI 생성), Bolt.new(풀스택 앱 생성) 등이 있습니다. Lovable과 유사한 도구로는 Bolt.new가 무료 크레딧이 넉넉해서 비교해볼 만합니다. 어떤 도구든 "문제를 정의하는 능력"이 결과 품질을 결정합니다. 도구보다 프롬프트가 중요합니다.
</Toggle>

<Toggle title="오늘 만든 앱, 실제로 팀에 배포해도 될까요?">
충분히 가능합니다. 단, 배포 전에 ① 실제 데이터로 테스트, ② 엣지 케이스(예외 상황) 확인, ③ 팀원 피드백 수렴 이 세 단계를 거치길 권장합니다. "완성된 제품"이 아닌 "팀 내 실험용 도구"로 시작하면 심리적 부담이 줄어듭니다.
</Toggle>

# 스텝 5: 데모타임 + 회고

## 데모타임

각 조가 만든 앱을 공유합니다.

<Callout type="default">
**오늘 아침에는 아이디어만 있었습니다. 지금은 동작하는 URL이 있습니다.**
완성도가 아닌, **이 변화 자체**가 데모의 핵심입니다.
</Callout>

### 데모 형식

- 조별 발표: 아래 3가지를 공유합니다
  1. **Before**: "우리 팀은 ___를 매번 수작업으로 했습니다"
  2. **After**: 배포된 URL에서 핵심 기능을 라이브 시연
  3. **발견**: "만들면서 알게 된 것은 ___입니다"
- 나머지 조: 발표 조의 **URL에 직접 접속**해서 써보기
- 피드백: "이 앱을 내 업무에 쓴다면 ___가 있으면 좋겠다" 1가지

<Callout type="warning">
**데모가 안 될 때 대응 가이드**

기술적인 문제는 언제든 생깁니다. 당황하지 말고 아래 순서로 대응하세요.

- **배포 URL이 안 열릴 때** → Lovable 에디터의 로컬 프리뷰(오른쪽 패널)로 시연합니다. 배포가 안 됐을 뿐, 앱은 동작합니다.
- **앱이 에러 화면을 보일 때** → 에러 전/후 스크린샷을 보여주세요. "이렇게 만들었고, 이 부분을 개선해야 한다는 걸 발견했다"는 것 자체가 발표 내용입니다.
- **아예 미완성일 때** → 프롬프트 대화 내역과 설계 과정을 발표합니다. "우리가 해결하려 한 문제, 어떤 접근을 시도했는지"를 공유하면 됩니다.

완성도가 아닌 **사고 과정**이 이 워크숍의 핵심입니다.
</Callout>

## 회고

### 개인 회고

아래 3가지에 답해보세요. 조원과 돌아가며 공유합니다.

1. **Before & After**: 오늘 시작 전의 나 vs 지금의 나, 가장 달라진 점은?
2. **Best Moment**: 오늘 가장 뿌듯했거나 놀라웠던 순간은?
3. **Monday Action**: 월요일에 바로 시도해볼 수 있는 것 하나는?

### 도구 가이드

오늘 사용한 도구들의 역할을 정리합니다.

| 단계 | 도구 | 역할 |
|------|------|------|
| 아이디어 → 프로토타입 | **Lovable** | 대화로 동작하는 앱을 만든다 |
| 데이터 저장 | **Lovable Cloud** | 새로고침해도 데이터가 유지된다 |
| 정밀한 개선 | **Claude Code** (VS Code) | 코드 수준으로 기능을 추가·수정한다 |
| 배포 | **Lovable Publish** | 다른 사람이 접속할 수 있는 URL을 만든다 |
| 자산화 | **CLAUDE.md** + 커스텀 커맨드 | 반복 작업을 자동화하고 팀과 공유한다 |

### Q&A

**Q. 회사 내부 데이터를 Lovable이나 Claude에 넣어도 되나요?**

개인정보·비밀 데이터를 외부 AI 서비스에 직접 입력하는 것은 현재 회사 보안 정책을 먼저 확인해야 합니다. 오늘 워크숍에서는 가상/샘플 데이터로 구조만 만들고, 실제 운영 시 사내 폐쇄망 환경이나 API 방식으로 전환하는 것이 현실적인 경로입니다.

**Q. 무료로 계속 쓸 수 있나요? 비용은 어떻게 되나요?**

Lovable은 무료 플랜(월 5개 프로젝트)과 유료 플랜(월 $25~)이 있습니다. Claude는 무료 플랜도 있지만 Claude Code를 쓰려면 Pro 구독($20/월)이 필요합니다. 처음엔 무료로 충분히 검증하고, 실무에 반복 사용한다면 유료 전환을 검토하세요. 수작업 시간 대비 ROI를 따지면 대부분 충분히 납득할 수준입니다.

**Q. 만든 앱을 팀원들이 쓰려면 어떻게 하나요?**

Lovable에서 **Publish** 버튼을 누르면 누구나 접속할 수 있는 URL이 생깁니다. 별도 서버 설치 없이 링크만 공유하면 됩니다. 다만 Lovable 무료 플랜은 공개 URL 수에 제한이 있으니, 팀 전체가 쓰는 앱이라면 유료 플랜이나 별도 배포(Vercel 등)를 검토하세요.

**Q. 코딩을 몰라도 나중에 유지보수할 수 있나요?**

기능 추가·수정은 Lovable 채팅으로 계속 할 수 있습니다. 다만 버그가 생기거나 복잡한 로직이 필요해지면 코드를 봐야 하는 순간이 옵니다. Claude Code를 이용하면 코드를 몰라도 "이 버튼이 동작을 안 해"라고 설명하는 것만으로 수정이 가능합니다. 완전한 무코딩은 아니지만, 개발자 의존도는 크게 낮출 수 있습니다.

**Q. 회사 기존 시스템(그룹웨어, ERP 등)과 연동하려면 어떻게 하나요?**

두 가지 경로가 있습니다. ① 시스템이 API를 제공한다면 Claude Code로 연동 코드를 작성할 수 있습니다. ② API가 없다면 엑셀/CSV 내보내기 → 앱에 업로드하는 방식으로 "반쪽짜리 자동화"도 충분히 효과적입니다. 오늘 만든 앱 대부분이 이 방식입니다.

**Q. Claude Code 말고 다른 AI 코딩 도구는 없나요?**

GitHub Copilot(에디터 자동완성), Cursor(AI 네이티브 에디터), v0(Vercel, UI 생성), Bolt.new(풀스택 앱 생성) 등이 있습니다. Lovable과 유사한 도구로는 Bolt.new가 무료 크레딧이 넉넉해서 비교해볼 만합니다. 어떤 도구든 "문제를 정의하는 능력"이 결과 품질을 결정합니다. 도구보다 프롬프트가 중요합니다.

**Q. 오늘 만든 앱, 실제로 팀에 배포해도 될까요?**

충분히 가능합니다. 단, 배포 전에 ① 실제 데이터로 테스트, ② 엣지 케이스(예외 상황) 확인, ③ 팀원 피드백 수렴 이 세 단계를 거치길 권장합니다. "완성된 제품"이 아닌 "팀 내 실험용 도구"로 시작하면 심리적 부담이 줄어듭니다.
